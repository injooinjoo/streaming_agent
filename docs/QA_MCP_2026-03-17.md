## Findings

### P1 blocker - `/api/auth/me` returns 404 in live path, so `overlayHash` never reaches the settings UI

- Repro:
  1. Open `http://127.0.0.1:4173/`
  2. Open `http://127.0.0.1:4173/streaming-agent`
  3. Enter chat settings
  4. `URL 복사` and `새창으로 열기` remain disabled
- Evidence:
  - `.playwright-cli/network-2026-03-17T10-10-36-183Z.log`
  - `.playwright-cli/network-2026-03-17T10-13-13-051Z.log`
  - `.playwright-cli/page-2026-03-17T10-12-01-923Z.yml`
- Observed behavior:
  - Frontend requests `GET /api/auth/me` with `Bearer auto-login-token`
  - Backend responds `404`
  - Frontend falls back to the local demo user, but `userHash` stays null
  - All user-hash-based overlay actions remain disabled
- Relevant code:
  - `client/src/contexts/AuthContext.jsx`
  - `server/middleware/auth.js`
  - `server/routes/auth.js`
  - `client/src/components/settings/ChatSettings.jsx`
  - `client/src/components/settings/shared/SettingsHeader.jsx`
- Root-cause hypothesis:
  - The frontend expects `/api/auth/me` to enrich the demo user with `overlayHash`
  - The backend auto-login path injects a fixed dev user payload, then `/auth/me` immediately looks up `req.user.id`
  - In the tested environment the backend behaves like mock/fallback mode, so the lookup does not resolve a persisted user row and returns `404`
  - Result: save works, but user-specific overlay URLs cannot be issued

- Evidence:
  - `.playwright-cli/console-2026-03-17T10-12-48-932Z.log`
- Relevant code:
